# ZiTag / ZiChecker — фронтенд для работы с VAST

Одностраничное веб-приложение **ZiTech**: разбор рекламного **тега** (URL) и глубокая проверка **VAST XML** — без отдельного бэкенда бизнес-логики, только статика и минимальный HTTP-сервер для раздачи файлов.

## Стек

| Компонент | Назначение |
|-----------|------------|
| `index.html`, `style.css` | Разметка и стили, вкладки ZiTag и ZiChecker |
| `app.js` | Логика ZiTag: парсинг URL, таблица параметров, валидация, темы |
| `checker.js` | Логика ZiChecker: парсинг VAST, wrapper-цепочка, плеер, отчёты |
| `ad-infra.js` | Каталог доменов SSP/DSP и эвристики по `<AdSystem>` |
| `server.js` | Node HTTP — раздача статики (порт по умолчанию **3847**) |
| CodeMirror 5 | Редактор XML во вкладке ZiChecker (CDN) |

Проверка типов: `npm run check` (TypeScript, `jsconfig.json`).

## Запуск

```bash
npm start
```

Откройте в браузере: `http://localhost:3847/` (порт задаётся переменной окружения `PORT`). Прямые пути приложения на сайте: [zi-tech.ru/digital/zitag](https://zi-tech.ru/digital/zitag) (ZiTag) и [zi-tech.ru/digital/zichecker](https://zi-tech.ru/digital/zichecker) (ZiChecker); локально то же без домена: `/digital/zitag`, `/digital/zichecker`.

### Docker

```bash
docker build -t zitag .
docker run -p 3847:3847 zitag
```

## ZiTag (вкладка «Парсер»)

**Назначение:** разобрать **строку VAST-тега** как URL — выделить **базовый URL (BASE)** без query-строки и таблицу параметров `ключ=значение`. Запрос к рекламному серверу **не выполняется**.

**Возможности:**

- Парсинг вставленного тега; редактирование BASE и параметров в таблице (вкл/выкл строки, добавление, сортировка ключей A–Я).
- Типы значений: макросы в `{…}`, `[…]` (IAB), `%…%` (в т.ч. после URL-decode, например `%25…%25`), полные `https://…`, статика, пустые значения.
- Подсказки по распространённым макросам и именам query-параметров (GAM, OpenRTB-стиль и др.).
- Валидация структуры тега с выводом ошибок и предупреждений.
- Сборка итогового URL внизу страницы; копирование URL и варианта с кодированием.
- Блок **«Инфраструктура»:** сопоставление домена BASE и полных `https://…` в значениях параметров с внутренним каталогом (`ad-infra.js`).
- История недавних тегов в `localStorage`.
- Счётчики: параметры, макросы, отключённые строки, статус проверки.
- Внизу вкладки — **справка парсера** (структура URL, макросы IAB/GAM/OpenRTB, ориентиры по платформам).

## ZiChecker (вкладка «Валидатор»)

**Назначение:** загрузить или вставить **VAST XML**, разобрать документ, при необходимости **раскрутить цепочку wrapper’ов**, просмотреть медиа и трекеры, **проиграть** линейное видео и увидеть срабатывание событий.

**Режимы ввода:**

- **По URL** — HTTP(S)-загрузка VAST (с диагностикой).
- **По XML** — вставка XML в редактор: форматирование, проверка well-formed.

**Возможности:**

- Разбор `<VAST>`: версия, InLine / Wrapper, `AdSystem`, `AdTitle`, `Duration`, `skipoffset`, события (Impression, Error, Tracking и др.).
- Рекурсивное разрешение **wrapper-цепочки** до InLine или ошибки.
- Список **MediaFile** и нелинейных/интерактивных ресурсов; детекция VPAID/SIMID; подсказки по совместимости.
- Встроенный **видеоплеер**, выбор источника, имитация skip / click / fullscreen, лента событий по трекерам; при VPAID/SIMID — заметка над плеером (линейное видео — отдельные `MediaFile`).
- Валидация по правилам IAB и распространённым требованиям SSP (обязательные элементы, длительность, длина `VASTAdTagURI`, delivery и т.д.).
- Сводка по документу, блок диагностики («почему не играет»), проверки compliance (где реализовано).
- **Инфраструктура:** те же карточки платформ по доменам из URL в ответе и по тексту `<AdSystem>`.
- Под блоком ввода XML — **справка валидатора** (версии VAST, `Tracking`, `MediaFile`, коды `Error`, IMA, OMID/SIMID, CTV).

| Модуль | Роль |
|--------|------|

### Сообщения compliance и консоли

В сводке остаются в основном **проверки по вашему конкретному VAST** (IAB, HTTP, отсутствие квартилей, только WebM на CTV и т.д.). Общие ориентиры по **IMA, CTV, VPAID+MP4, SIMID, NonLinear, версиям VAST** и смыслу блока **«Платформы»** вынесены во **встроенную справку валидатора** (разделы «Google IMA SDK», «CTV…», «Инфраструктура…», «Версии VAST», «MediaFile»).

| Источник | Типичный смысл |
|----------|----------------|
| **IAB VAST** | Несоответствие спецификации или типичные требования бирж (квартали, Duration, ClickThrough и т.д.). |
| **Google IMA** | В первую очередь **HTTP / mixed content** на HTTPS-страницах. |
| **CTV / OTT** | Жёсткие случаи для ТВ: только VPAID без MP4, только WebM, низкий битрейт; подробности по соседству VPAID и MP4 — в справке. |
| **Mobile** | **HTTP** под iOS ATS / политики Android (дублирует критичность mixed content для приложений). |
| **Платформы** | Краткий список распознанных по доменам/AdSystem имён без длинного пояснения — см. справку «Инфраструктура…». |

**Консоль браузера:** предупреждение Chrome про `iframe` с `sandbox="allow-scripts allow-same-origin"` означает, что такая комбинация **ослабляет** изоляцию (это известное поведение движка). Для песочницы VPAID она оставлена осознанно: иначе часть сценариев (в т.ч. загрузка скрипта и blob-прокси) может не сработать. Сообщения вроде `content.js` и «message port closed» чаще всего идут от **расширений браузера**, а не от этого репозитория.

### Песочница VPAID (ZiChecker)

Цепочка вызовов в iframe: **`getVPAID`** / **`getVPAIDAd`** (лоадер Яндекса) → **handshakeVersion** → **initAd** → **AdLoaded** → **`startAd`**. У лоадера Яндекса внутренний **VpaidPlayer** поднимается асинхронно, поэтому **`startAd` повторяется с задержкой до готовности** (иначе типична ошибка `VPAID_PLAYER_LOADER_ACTION_METHOD_CALLED_WITHOUT_VPAID`).

Режим **«По URL»**: для относительных путей в `MediaFile` используется **база** ответа; при **mixed content** (HTTP-скрипт на HTTPS-странице) — **прокси** через blob. Внешние ограничения: **CSP** на странице, **X-Frame-Options** / политики встраивания у CDN — скрипт может не загрузиться в песочнице.

Если в статусе песочницы всё же появляется **`startAd-error: VAS Error [VPAID_PLAYER_LOADER_ACTION_METHOD_CALLED_WITHOUT_VPAID][DEFAULT_ERROR_MESSAGE]`** после серии повторов — креатив или сервер может не отдавать рабочий VPAID в этом окружении; предупреждение Chrome про **sandbox** (см. выше) на это не влияет.
